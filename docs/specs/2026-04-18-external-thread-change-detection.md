# External Thread Change Detection Spec

## Relationship To Existing Specs

This spec extends:

- `docs/specs/2026-04-18-realtime-session-list.md` — the global SSE channel and
  session list derivation

It addresses a gap identified during testing:

- The global SSE channel only relays events from threads the bridge has subscribed
  to via `resumeThread`. Events generated by external processes (Codex TUI, CLI,
  other app-server instances) are invisible because each Codex process runs an
  independent app-server instance.

Background on the app-server architecture:

- `docs/reference/2026-04-18-codex-app-server-internals.md`

## Background

The realtime session list feature uses a global SSE channel to push thread
status updates to connected clients (mobile app, desktop browser). Events reach
global clients from two sources:

1. **Per-thread subscriptions**: the bridge subscribes to threads via
   `thread/resume` when a user opens a thread detail view. These events flow
   to both per-thread and global SSE clients via `broadcast()`.
2. **ThreadChangeDetector** (this spec): watches the shared filesystem for
   external changes.

Source 1 covers threads the bridge is actively interacting with. Source 2 covers
everything else — threads modified by the Codex TUI, CLI, or any other process
that writes to the shared `codexHome/sessions/` directory.

The root cause of the gap (documented in
`docs/reference/2026-04-18-codex-app-server-internals.md`):

1. The Codex TUI runs an **embedded in-process app-server** (same PID, in-memory
   MPSC channels)
2. The bridge spawns a **separate `codex app-server` child process** (stdio
   JSON-RPC)
3. These two instances share the same `codexHome` filesystem but have
   **independent in-memory state and event systems**
4. `thread/resume` subscriptions only work within a single app-server process

Both instances share the same filesystem. The Codex app-server writes thread
state to rollout files (`{codexHome}/sessions/YYYY/MM/DD/rollout-*.jsonl`), and
`thread/list` scans the filesystem each time. This shared filesystem is the only
mechanism for cross-instance state observation.

## Protocol Changes

Since the project has no users yet, we take this opportunity to improve the
`BridgeEvent` protocol:

### `threadStarted.thread`: `ThreadDetail` → `ThreadSummary`

Currently `threadStarted` carries a full `ThreadDetail` (including turns,
settings, contextUsage). The SDK immediately converts it to `ThreadSummary` via
`toThreadSummary()`, discarding the extra fields. This conversion is wasteful
and prevents the detector from synthesizing valid `threadStarted` events (since
`listThreads()` only returns `ThreadSummary`).

Change the protocol to use `ThreadSummary` directly:

```typescript
// Before
| { type: 'threadStarted'; threadId: string; thread: ThreadDetail }

// After
| { type: 'threadStarted'; threadId: string; thread: ThreadSummary }
```

The bridge's event translation layer converts `ThreadDetail` to `ThreadSummary`
before emitting `threadStarted`. The SDK's `updateThreadSummaryState()` no
longer needs `toThreadSummary()` — it uses the summary directly.

This does NOT affect the thread detail view. When a user opens a thread,
`resumeThread()` returns `ThreadDetail` through a separate JSON-RPC call — this
flow is independent of `threadStarted` SSE events.

### New `threadDeleted` event type

The detector can detect when a thread disappears from `listThreads()` results.
Currently there is no protocol event for this, so deleted threads remain visible
indefinitely.

```typescript
| { type: 'threadDeleted'; threadId: string }
```

The SDK's `updateThreadSummaryState()` removes the thread from the list when
this event is received.

### SDK detail-state reducer (`applyThreadEvent`)

The SDK's `applyThreadEvent()` function handles events for the per-thread
detail view. It currently has a `threadStarted` case that returns
`event.thread` (a `ThreadDetail`). After the protocol change, `event.thread`
becomes `ThreadSummary`, which is incompatible.

Since the detail view loads full thread data via `readThread()` (not SSE
`threadStarted` events), both `threadStarted` and the new `threadDeleted`
should return the current thread unchanged in `applyThreadEvent()`. The detail
view handles its own navigation lifecycle.

## Goals

- Detect thread state changes caused by external Codex processes (TUI, CLI,
  other app-server instances) without requiring changes to user workflow
- Forward detected changes to connected global SSE clients using the existing
  event pipeline
- Minimize performance overhead — zero cost when no global SSE clients are
  connected
- Keep the SDK and client unchanged (except for the protocol changes above)
- No bulk subscription of all threads via `resumeThread` — expensive and still
  misses TUI events

## Non-Goals

- Replacing the existing per-thread SSE subscription model
- Streaming real-time agent deltas or reasoning events for external turns
  (only summary-level status changes)
- Modifying the Codex TUI or upstream app-server
- Supporting cross-instance turn streaming (only status, name, new-thread, and
  deletion detection)
- Detecting `pendingRequests` changes for external threads (requires parsing
  JSONL rollout files — deferred)

## Scope

### Included

- Protocol changes: `threadStarted` uses `ThreadSummary`, new `threadDeleted`
  event type
- Filesystem-based change detection on the `{codexHome}/sessions/` directory
- Diff of full `ThreadSummary` snapshots via `listThreads`
- Synthesis of `threadStatusChanged`, `threadNameUpdated`, `threadStarted`,
  and `threadDeleted` events for detected changes
- Integration with the existing `ThreadEventStreamRegistry` global SSE broadcast
- Lifecycle management tied to global SSE client presence

### Deferred

- `pendingRequests` change detection for external threads
- Preview text change detection (requires parsing JSONL)
- Cross-instance turn streaming (deltas, items, reasoning)
- Archived thread detection (`archived_sessions/` directory)
- Integration with the Codex TUI to share a single app-server instance

## Data Flow

### Current (Broken for External Changes)

```
Codex TUI (embedded app-server)
  |  modifies rollout files
  v
Shared Filesystem (codexHome/sessions/)
  |  NOT watched by bridge
  v
(nothing happens — mobile client never sees the change)
```

### Proposed

```
Codex TUI (embedded app-server)
  |  modifies rollout files
  v
Shared Filesystem (codexHome/sessions/)
  |  chokidar detects .jsonl changes
  v
ThreadChangeDetector
  |  debounce 500ms
  |  listThreads() -> diff against cache (full ThreadSummary)
  |  synthesize events:
  |    - new thread       -> threadStarted (with ThreadSummary)
  |    - status changed   -> threadStatusChanged
  |    - name changed     -> threadNameUpdated
  |    - thread removed   -> threadDeleted
  |    - updatedAt only   -> threadStatusChanged (recency/sorting)
  v
ThreadEventStreamRegistry.broadcast()
  |  existing global SSE pipeline
  v
Mobile / Desktop Clients
```

## Technical Details

### Change Detection Mechanism

Use `chokidar` to watch `{codexHome}/sessions/` recursively:

- **Filter**: Only `.jsonl` files (rollout files)
- **Events**: `add`, `change`, `unlink` (file create, modify, delete)
- **Ignore initial**: Do not trigger on watcher startup (existing files)
- **Debounce**: 500ms to batch rapid writes during active turns

### Cache and Diff Algorithm

The detector caches full `ThreadSummary` objects, not just status/updatedAt:

```typescript
// Cache stores complete ThreadSummary for each thread
#cache = new Map<string, ThreadSummary>();
```

On each debounced change:

1. Call `threadService.listThreads({})` — scans filesystem via app-server
2. Build a `Map<threadId, ThreadSummary>` from the result
3. Compare with previously cached snapshot:
   - **New thread** (in new, not in cache) → `threadStarted` with full summary
   - **Changed status** → `threadStatusChanged` event
   - **Changed name** → `threadNameUpdated` event
   - **Changed updatedAt only** → `threadStatusChanged` with current status
     (covers preview/recency updates)
   - **Thread removed** (in cache, not in new) → `threadDeleted` event
4. Replace `#cache` with new map
5. Forward each synthesized event through `broadcast()`

### Lifecycle

The detector is active only when global SSE clients are connected:

- **Start**: When the first global SSE client connects (in `addGlobalClient()`)
  - Create the chokidar watcher first (with `ignoreInitial: true`) so no
    filesystem changes are missed during the async gap
  - Then call `listThreads()` to populate the cache (no events emitted)
  - Run a post-startup diff (`scheduleHandleChange()`) to catch any changes
    that occurred during the async gap while debounce was suppressed by
    `#starting = true`
- **Stop**: When the last global SSE client disconnects (in
  `removeGlobalClient()`)
- **Always stop**: On bridge shutdown (in `close()`)

This ensures zero overhead when no mobile/desktop clients are viewing the
session list.

Cache initialization prevents spurious events: on first start, all existing
threads are loaded into the cache silently. Only changes detected AFTER this
baseline trigger events.

### Event Shapes

Synthesized events use `BridgeEvent` types (with the protocol changes above):

```typescript
// New thread detected — carries full ThreadSummary
{ type: 'threadStarted', threadId, thread: ThreadSummary }

// Thread status or recency changed
{ type: 'threadStatusChanged', threadId, status: ThreadRuntimeStatus }

// Thread renamed
{ type: 'threadNameUpdated', threadId, threadName: string | null }

// Thread deleted or archived
{ type: 'threadDeleted', threadId }
```

The SDK's `updateThreadSummaryState()` handles all of these:

- `threadStarted` → upsert into thread list (now uses `ThreadSummary` directly,
  no `toThreadSummary()` conversion needed)
- `threadStatusChanged` → update status field
- `threadNameUpdated` → update name field
- `threadDeleted` → remove from thread list (new handler)

### Error Handling

- If `listThreads()` fails, log and skip — do not disrupt the watcher
- If the watch target directory does not exist, log a debug message and skip
  (no sessions yet)
- If chokidar reports an error, log and continue (FSEvents is self-healing)
- If `listThreads()` fails repeatedly, the debounced handler continues to retry
  on the next filesystem event (no exponential backoff in V1)

## Known Limitations

- **Debounce masking**: rapid status transitions (e.g., `idle → active → idle`
  within 500ms) may be missed. The detector only sees the final state after
  debounce. This is acceptable for V1 — the session list will show the correct
  final state.
- **No `pendingRequests` detection**: `pendingRequests` come from the app-server's
  in-memory state, not the filesystem. External thread pending requests will not
  be detected. The mobile user will see the pending request only when the bridge's
  own app-server emits it (which won't happen for external threads).
- **No preview text updates**: preview comes from the first user message in the
  rollout file, which requires parsing JSONL. The detector only compares
  summary-level fields.
- **No archived thread detection**: archived threads move to `archived_sessions/`,
  which is a separate directory not watched by the detector.
- **No thread deletion from TUI**: if the TUI archives (not deletes) a thread,
  it will appear removed from the session list (no longer in `listThreads()`
  results) but the `threadDeleted` event is correct for this case.

## Acceptance Criteria

1. Mobile app shows thread status changes within 2 seconds of TUI activity
2. New threads created via TUI appear in the mobile session list within 2 seconds
3. Thread renames via TUI propagate to mobile within 2 seconds
4. Threads deleted via TUI disappear from the mobile session list within 2 seconds
5. No CPU or I/O overhead when no global SSE clients are connected
6. Existing per-thread SSE streaming (agent deltas, reasoning, command output)
   is unaffected
7. Existing global SSE events from per-thread subscriptions are unaffected
8. `pnpm --filter @my-codex-app/bridge typecheck` passes
9. `pnpm --filter @my-codex-app/bridge test` passes
10. `pnpm fmt` produces no changes
